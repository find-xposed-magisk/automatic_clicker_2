# 激活码功能设计

## 1. 目标

为程序增加“按月激活”能力。用户每月首次启动时必须输入当月激活码；校验通过后，本月内无需重复输入。若激活码缺失、过期或错误，则弹出 `QMessageBox` 提示“当前激活码已经到期”，用户点击确认或关闭后，程序立即退出。

## 2. 总体约束

- 禁止使用本地系统时间作为激活依据。
- 当前时间必须通过独立函数 `get_webserver_time()` 从 NTP 服务器获取。
- 激活信息写入 SQLite 数据库的“设置”表，不写入 `config.ini`。
- 激活相关逻辑全部集中在一个独立 Python 文件中，其它文件只负责导入并调用。

## 3. 时间来源方案

### 3.1 获取方式

使用如下函数作为唯一时间入口：

```python
def get_webserver_time() -> datetime:
    """获取网络时间，如果网络异常则返回None"""
    # NTP服务器列表，按优先级排序
    ntp_servers = [
        'ntp.aliyun.com',
        'time.windows.com',
        'pool.ntp.org',
        'time.nist.gov',
    ]
    for server in ntp_servers:
        for retry in range(3):  # 每个服务器最多重试3次
            try:
                c = ntplib.NTPClient()
                r = c.request(server, version=3, timeout=3)
                ntp_time = datetime.datetime.fromtimestamp(r.tx_time)
                print(f"成功从 {server} 获取NTP时间: {ntp_time.date()}")
                return ntp_time.date()
            except Exception as e:
                print(f"第{retry + 1}次从 {server} 获取NTP时间失败: {e}")
                if retry == 2:
                    print(f"从 {server} 获取时间的所有重试都失败")
                continue

    print("所有NTP服务器都无法获取时间")
    return None
```

### 3.2 使用规则

- 所有激活校验、月份判断、激活码生成，均以 `get_webserver_time()` 返回值为准。
- 如果返回 `None`，按“校验失败”处理，不允许进入主界面。
- 推荐错误提示单独区分为“无法获取网络时间”或统一视为激活失败；本期可统一退出，后续再细化提示文案。

## 4. 激活码规则

### 4.1 月份基准值

- 取网络时间所在月份的第一天作为基准日期。
- 推荐标准字符串格式：`YYYY-MM-01`
- 示例：
  - 若 NTP 返回日期为 `2026-03-09`，则基准值为 `2026-03-01`

### 4.2 生成公式

- 原始串：`{month_start}|{secret_key}`
- 示例：`2026-03-01|YOUR_SECRET_KEY`
- 对原始串执行不可逆摘要算法，推荐 `SHA-256`
- 可将摘要结果转为大写十六进制，再截取前 16 位作为激活码

## 5. 数据库存储方案

### 5.1 存储位置

激活信息写入 `命令集.db` 的“设置”表，不再写入 `config.ini`。

### 5.2 推荐字段设计

若“设置”表已存在通用键值结构，则直接复用；若没有，则扩展为如下逻辑字段：

- `激活月份`：例如 `2026-03`
- `激活状态`：`True` / `False`
- `最后校验时间`：网络时间字符串，例如 `2026-03-09`

### 5.3 校验规则

- 当“设置”表中的 `激活月份` 与当前网络月份一致，且 `激活状态=True` 时，允许进入程序。
- 进入新月份后必须重新输入激活码。
- 数据缺失、记录异常、数据库读写失败时，一律按未激活处理。

## 6. 模块化设计

所有激活相关代码只放在一个独立文件中，建议文件名为：

- `activation_service.py`
  或
- `激活服务.py`

建议由该模块独立完成以下职责：

- 获取网络时间 `get_webserver_time()`
- 计算当月第一天 `get_month_start()`
- 生成激活码 `generate_activation_code()`
- 读取数据库激活状态 `get_activation_state()`
- 保存数据库激活状态 `save_activation_state()`
- 启动总校验 `check_activation_or_exit()`
- 弹出输入框 `prompt_activation_code()`
- 失败后提示并退出 `show_expired_and_exit()`

其他文件只允许这样使用：

- `Clicker.py` 在启动阶段导入并调用 `check_activation_or_exit()`
- 不在 `Clicker.py`、`ini控制.py`、`数据库操作.py` 中分散编写激活业务逻辑

## 7. 启动流程

1. 程序启动。
2. `Clicker.py` 导入激活模块。
3. 在创建 `Main_window()` 前调用 `check_activation_or_exit()`。
4. 激活模块先调用 `get_webserver_time()` 获取网络日期。
5. 读取数据库“设置”表中的激活记录。
6. 若当前月已激活，则直接继续启动。
7. 若未激活，则弹出激活码输入框。
8. 输入正确则写入“设置”表并继续启动。
9. 输入错误、取消、关闭输入框、无法获取网络时间、数据库异常时，弹出“当前激活码已经到期”。
10. 用户关闭消息框后执行 `sys.exit()`。

## 8. UI 方案

- 输入方式优先使用 `QInputDialog.getText()`，实现成本最低。
- 输入框标题：`激活验证`
- 输入框提示：`请输入本月激活码：`
- 失败提示框：

```python
QMessageBox.critical(None, "提示", "当前激活码已经到期")
```

## 9. 数据访问建议

虽然激活业务逻辑必须集中在单独模块中，但该模块内部可以直接使用 `sqlite3` 操作 `命令集.db`，避免把激活细节分散到其它模块。

如后续希望统一数据库入口，可由激活模块内部调用 `DatabaseOperation`，但不建议把激活专用逻辑再拆回 `数据库操作.py`。

## 10. 依赖与风险

- 需新增依赖：`ntplib`
- 网络不可用时，程序将无法通过激活校验
- 若设备无法访问上面的 NTP 服务，会导致无法启动
- 若后续要降低误杀率，可补充 HTTP 时间源作为 NTP 失败后的降级方案

## 11. 实施顺序

1. 新建独立激活模块文件，实现 `get_webserver_time()`、激活码计算、数据库读写、退出控制。
2. 为数据库“设置”表补齐激活相关键值。
3. 在 `Clicker.py` 启动主流程中接入 `check_activation_or_exit()`。
4. 验证以下场景：
   - 当月已激活可直接进入
   - 当月首次输入正确可进入
   - 输入错误后弹框并退出
   - 关闭输入框后弹框并退出
   - 所有 NTP 服务失败后弹框并退出
